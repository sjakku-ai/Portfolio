Building a GPT From Scratch, Part 4: Live URL Fetch and Clean

Every session so far fed the model a prompt I typed by hand. This one wires in real-world input: give it a URL, and the script fetches the page, extracts just the article text out of all the surrounding HTML noise, and builds a summarization prompt that respects GPT-2’s 1024-token context window.

The goal for this session was narrow on purpose: take the greedy-decode loop from Part 3 and give it something to actually chew on — a real article, not a hardcoded string. That meant three new problems to solve: downloading a page, pulling the article out of it, and making sure the result actually fits in the model’s head.

Fetching and cleaning: two libraries, two jobs

It’s tempting to reach for one tool that does “get me the article from this URL,” but that’s really two separate problems, and fetch_and_clean() keeps them separate on purpose:

headers = {"User-Agent": "Mozilla/5.0 (compatible; MiniGPT-learning-script/1.0)"}
response = requests.get(url, headers=headers, timeout=10)
response.raise_for_status()

article_text = trafilatura.extract(response.text, favor_precision=True)

if not article_text:
  raise ValueError(f"trafilatura could not extract article text from {url}")
  • requests does the HTTP GET — nothing more. It knows nothing about HTML structure; it just hands back whatever bytes the server sent, the same as curl would.
  • trafilatura takes that raw HTML and figures out which part of the page is actually “the article,” as opposed to the nav bar, ads, comments, and cookie banners around it. It parses the DOM, scores each region by text density and structural cues, and returns just the likely main content as plain text — sparing me from writing brittle BeautifulSoup selectors that break the moment a site changes its layout.

A few details earn their keep here: the User-Agent header avoids some sites’ basic bot-blocking, which would otherwise return an error page instead of the article. timeout=10 means a hung connection fails loudly instead of blocking the script forever. And raise_for_status() turns HTTP errors (404, 500…) into an exception immediately — without it, the script would happily keep going with an error page’s HTML as if it were real content. favor_precision=True tells trafilatura: when in doubt, leave text out rather than risk including boilerplate. For summarization, a little missing content is far less harmful than junk sneaking into the prompt.

Building the prompt

GPT-2 isn’t instruction-tuned the way ChatGPT is — it was only ever trained to predict “what word comes next” over a huge pile of raw internet text. It has no built-in concept of “please summarize this.” The trick is to construct text that looks like the start of a summary and let the model’s next-token statistics carry the pattern forward:

def build_prompt(article_text: str) -> str:
  instruction = "Summarize the following article in 3 sentences:\n\n"
  return f"{instruction}{article_text}\n\nSummary:"

That trailing "Summary:" is doing real work — it’s not decoration. It sets up a pattern GPT-2 has seen millions of times in its training data (article, then a labeled summary), so the most probable next tokens are ones that continue that pattern rather than, say, more article text.

Respecting the context window

GPT-2’s context window is 1024 tokens — and that’s not just a prompt limit, it’s a hard ceiling on prompt + generated tokens combined. A real article can easily blow past that on its own, so the prompt has to be truncated before it ever reaches the model, or generation fails outright.

token_ids = tokenizer.encode(prompt)
max_prompt_tokens = max_context_tokens - reserved_for_generation

if len(token_ids) <= max_prompt_tokens:
  return prompt

truncated_ids = token_ids[:max_prompt_tokens]
return tokenizer.decode(truncated_ids)

Two things worth calling out:

  1. The budget is split, not just capped. If the prompt is allowed to fill all 1024 slots, the model has zero room left to actually write a summary. reserved_for_generation (100 tokens by default) carves out space for the output before the prompt gets to claim anything.
  2. Truncating happens on tokens, not characters or words. Token count doesn’t map cleanly onto either — GPT-2’s tokenizer can split one rare word or name into several tokens, or fold short words together. Guessing a cutoff from character count risks being wildly off in either direction; only the tokenizer’s own encode() gives the real number, so slicing the token list and decoding back to text is the only way to land the cut exactly where intended.

Zoomed out, the whole pipeline is a series of hand-offs, each one narrowing what the model will actually see: full HTML page → trafilatura strips it to article text only → build_prompt wraps that text with the instruction → truncate_to_token_limit caps it at ~924 tokens. The model never sees raw HTML, and it never sees more than those ~924 tokens no matter how long the source page is — anything past that cutoff is silently dropped, which is why the function prints a warning when it triggers.

The catch: for an article as long as the GPT-2 Wikipedia page, that 924-token budget only covers the opening sections — not the whole thing. So the “summary” really only reflects the beginning of the article. That’s the real limitation of truncating from the front, and worth knowing before trusting the output.

Running it end to end

The full pipeline is just those three pieces chained together, feeding into the greedy_generate loop from Part 3:

article_text = fetch_and_clean(url)
prompt = build_prompt(article_text)
prompt = truncate_to_token_limit(tokenizer, prompt, max_context_tokens=1024, reserved_for_generation=100)

summary = greedy_generate(model, tokenizer, prompt, max_new_tokens=100)

Nothing about the decoding loop itself changed from Part 3 — the point of this session was entirely upstream of it: turning an arbitrary URL into something the model can safely consume. The output quality is still rough (GPT-2 is a small, non-instruction-tuned model doing its best impression of summarization), but the pipeline now takes real input instead of a string I typed myself, which is the actual milestone here.
