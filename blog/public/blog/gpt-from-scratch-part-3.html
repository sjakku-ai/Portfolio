Building a GPT From Scratch, Part 3: Loading GPT-2 and Writing My Own Greedy Decode Loop

With the softmax and temperature math from Part 2 in hand, I loaded a real pretrained GPT-2 and generated text two ways — once through Hugging Face’s built-in generate(), and once through a decode loop I wrote myself — to prove I actually understood what was happening between them.

Step one on my build order from Part 1 was: get GPT-2 and the Hugging Face tokenizer doing basic generation through my own greedy-decode loop, to validate the pipeline works end to end. This post covers the two files that did that — load_and_generate_gpt2.py, which loads the model and calls the library’s own generation function, and greedy_decode.py, which reimplements that same generation by hand.

load_and_generate_gpt2.py: loading a pretrained checkpoint

This script is the “hello world” of using a pretrained model, broken into deliberate steps:

  1. Load a tokenizer that matches the model. A model only understands numbers, and every pretrained model was trained against one specific tokenizer — using a different one would mean the same number means something different to the model than intended. AutoTokenizer.from_pretrained("gpt2") pulls the exact tokenizer GPT-2 was trained with.
  2. Load the pretrained weights. "gpt2" is a checkpoint name — a saved snapshot of a neural network’s learned parameters after training, hosted on the Hugging Face Hub. Rather than spending OpenAI-scale compute retraining GPT-2 from nothing, AutoModelForCausalLM.from_pretrained("gpt2") downloads those already-learned weights straight into a GPT-2-shaped model. “CausalLM” just means it was trained to predict the next token given everything before it — a text-generation model, not a classifier.
  3. model.eval(). Some layers, like Dropout, behave differently during training (randomly disabling neurons to prevent overfitting) than during real use. .eval() switches every such layer into inference mode — a standing habit any time you’re not actively training.
  4. Encode the prompt. tokenizer(prompt, return_tensors="pt") turns the text into a PyTorch tensor of token IDs, shaped (batch=1, sequence_length).
  5. Generate, inside torch.no_grad(). Gradients are only needed for training, to work out how to adjust weights via backpropagation. Since this is a forward pass only, tracking gradients would burn memory and compute for nothing — so it’s disabled.

The actual generation call is model.generate(**inputs, max_new_tokens=20, do_sample=False, pad_token_id=tokenizer.eos_token_id). max_new_tokens caps how much new text gets produced — without it the model could ramble until it hit its context limit. do_sample=False is the key setting: it means greedy decoding, always taking the single most probable next token, which is fully deterministic — the same prompt always produces the same output. (do_sample=True would instead sample from the probability distribution, weighted by likelihood, for more varied but non-deterministic text — this is where the temperature math from Part 2 actually plugs in.)

tokenizer = AutoTokenizer.from_pretrained("gpt2", token=hf_token)
model = AutoModelForCausalLM.from_pretrained("gpt2", token=hf_token)
model.eval()

prompt = "The capital of France is"
inputs = tokenizer(prompt, return_tensors="pt")

with torch.no_grad():
  output_ids = model.generate(
    **inputs,
    max_new_tokens=20,
    do_sample=False,  # greedy: always take the single most probable token
    pad_token_id=tokenizer.eos_token_id,
  )

generated_text = tokenizer.decode(output_ids[0], skip_special_tokens=True)

greedy_decode.py: the same thing, by hand

The point of this file is to strip away model.generate() and write the loop it’s hiding, one token at a time:

for _ in range(max_new_tokens):
  outputs = model(input_ids)
  logits = outputs.logits[:, -1, :]  # only the LAST position predicts what's next
  next_token_id = logits.argmax(dim=-1)  # greedy: take the highest-scoring token
  if next_token_id == eos_token_id: break
  input_ids = torch.cat([input_ids, next_token_id], dim=-1)  # re-feed everything so far

Three things stood out to me writing this:

  1. The model doesn’t just predict “the next token.” Every forward pass returns logits shaped (batch, sequence_length, vocab_size) — one full probability-distribution-worth of logits for every position in the sequence, not just the end. Only the last row, logits[:, -1, :], is the prediction for what comes after everything seen so far; the earlier rows predicted tokens that are already fixed in place from previous steps. It’s easy to assume the model outputs one token’s worth of information — it actually outputs a distribution per position, every single step.
  2. Generation re-feeds the whole sequence, every step. There’s no persistent internal “memory” carried between loop iterations here — each iteration hands the model the prompt plus everything generated so far, and the model recomputes from scratch. That’s the literal meaning of autoregressive: each new token is generated conditioned on all previous tokens, including the ones the model itself just produced.
  3. Greedy decoding is just argmax. All of the softmax/temperature machinery from Part 2 is, in this file, replaced by the simplest possible choice: skip the sampling step entirely and always take the single highest-logit token. That’s exactly what do_sample=False means in load_and_generate_gpt2.py’s library call — the two files are doing the identical thing, just one hides it behind an API and one exposes every step.

Running both against the same prompt and confirming they produce identical output was the actual validation step — if my hand-rolled loop and Hugging Face’s generate() disagreed, something in my understanding of the pipeline was wrong. They matched, which is the real payoff of building this by hand: not the code itself, but the confidence that I know what every line of a library call like model.generate() is actually doing underneath.
