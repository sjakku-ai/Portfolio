AI System Design: Part-1 - The Infrastructure Every AI Engineer Needs to Understand

Why the unglamorous plumbing of VMs, containers, and load balancers decides whether your model ever reaches users.

Why the unglamorous plumbing of VMs, containers, and load balancers decides whether your model ever reaches users.

There's a persistent fantasy in AI circles that the hard part is the model. Get the architecture right, tune the hyperparameters, hit good validation loss — and the rest is deployment detail. This fantasy gets engineers into trouble.

The infrastructure layer — compute, networking, storage, and communication — is where most AI projects die. Not because the models are bad, but because the systems around them weren't built for production reality: variable load, hardware failures, dependency hell, and the chaos of running software at scale.

Why AI Workloads Break Your Assumptions

If your mental model of infrastructure comes from stateless, CPU-bound web services that scale by adding identical containers, AI inference breaks nearly every assumption.

GPU dependency. A serving container needs specific GPU hardware, a compatible CUDA version, the right driver, and exact CUDA libraries — all version-locked. Mount the wrong driver and it crashes silently. This hardware-software coupling has no analog in web engineering.

Memory behavior. Loading a model into VRAM takes 10–30 seconds; default Kubernetes liveness probes will kill the container before it's ready. A 7B model in FP16 needs 14GB, so a single 24GB A10G barely fits one copy.

Batch sensitivity. In web services, low throughput means low cost. In GPU inference, an idle GPU is a money-burning disaster — production GPU utilization averages around 5%. You must batch aggressively or autoscale to zero between bursts.

Compute: VMs, Containers, and Where Each Wins

A virtual machine is a complete isolated computer; a hypervisor (ESXi, KVM, Hyper-V) carves the physical server into VMs, each with its own OS. Heavier (1–4GB overhead) but genuinely isolated. Cloud GPU instances are VMs — when you request a p3.8xlarge, you get a VM with a GPU passed through. The right call for long training runs and compliance isolation.

Containers virtualize the OS, not the hardware. They start in seconds, add almost no overhead, and are fully described in a Dockerfile. For AI, they solve dependency hell: the image pins the exact Python, PyTorch, and CUDA libraries. The catch — GPU containers need the host to have the NVIDIA Container Toolkit and compatible drivers. Images bundle CUDA libraries but not drivers. This trips up everyone the first time.

Kubernetes and the GPU Scheduling Problem

Once you have more than a handful of containers, you need an orchestrator. But "Kubernetes for AI" differs sharply from "Kubernetes for web APIs":

Gang scheduling. A distributed training job needs all 8 pods to start simultaneously or it wastes GPUs. Standard scheduling can't guarantee this; Volcano and NVIDIA's KAI Scheduler add gang awareness.

GPU topology. Two GPUs on the same NVLink bus transfer at 600GB/s; across PCIe, maybe 16GB/s. A topology-blind scheduler can place a job on GPUs that talk 38× slower than optimal.

Dynamic Resource Allocation (DRA). Kubernetes 1.34 (GA 2026) replaced the legacy Device Plugin framework, letting you declare rich requirements ("40GB+ VRAM, NVLink, MIG support"), not just "a GPU."

Utilization. 100 A100s at 5% utilization burns ~$500K/month idle. The fix: aggressive batching plus scale-to-zero autoscaling. KEDA triggers on queue depth — the correct signal for inference — rather than CPU load.

Networking: The Parts You Can't Skip

Every component — training servers, feature stores, model servers, monitors — is a node with an IP. DNS maps names to addresses, so your code references feature-store.internal instead of an IP that changes on restart. In Kubernetes, CoreDNS gives every Service a stable name that survives pod restarts.

A load balancer distributes requests across instances, which is what makes horizontal scaling real. The AI subtlety: not all instances are equal. A pod still loading a 14GB model isn't ready even though it's "running." Readiness probes distinguish "running" from "ready to serve" — critical when loading takes 20+ seconds. Session affinity (routing a user to the same pod) can boost cache hit rates when a pod has already cached that user's embeddings.

REST vs. gRPC: An Honest Decision

Use REST for externally-facing APIs, large text/JSON payloads, easy debugging, and HTTP caching. Use gRPC for internal calls, dense numerical data (embeddings, tensors), and streaming — essential for LLM token streaming. A bonus often missed: gRPC's .proto files are a compile-time contract, so a schema change breaks the client at build time, not in production.

Storage: Match the Type to the Access Pattern

Object storage is the backbone — cheap and infinite, but 10–100ms latency makes it fine for batch, terrible for real-time serving. Block storage behaves like a local disk (sub-ms) for databases and caches. File storage offers a shared filesystem multiple training nodes can mount at once.

The Vector Database Question

Vector databases store embeddings and do approximate nearest-neighbor (ANN) search — finding semantically similar vectors without scanning everything. Algorithms like HNSW and IVF trade a little recall for huge speed: 10 nearest neighbors among 100M vectors in under 10ms. Choosing in 2026:

  • Pinecone — fully managed, fastest to ship; higher cost at scale.
  • Qdrant — open-source, self-hostable, high performance at lower cost.
  • pgvector — adds vectors to Postgres; pragmatic under ~5M vectors if you already run Postgres.
  • Weaviate — strong for hybrid vector + attribute filtering and multi-tenancy.

Caching: Three Distinct Layers

Conflating these leads to bad design. A feature cache (Redis/Memcached) holds slow-changing user/item features for single-digit-ms lookups. An embedding cache avoids re-embedding stable content. A response cache stores complete outputs for deterministic queries (FAQ-style), but not personalized or time-sensitive ones.

Architecture: The Answer Depends on Team Size

Start with a modular monolith — seriously. Microservices solve an organizational problem (many engineers deploying independently), not a technical one. Distributed systems add real complexity: network calls, serialization, service discovery, partial failures. A modular monolith with clean internal seams (ingestion, features, inference, monitoring) deploys as one unit while keeping the boundaries that allow future extraction. Shopify and Stack Overflow served millions as monoliths.

Split into services only when components have dramatically different scaling needs, separate deployment cadences, or need to be shared across products.

Kafka as the System's Spine

Kafka is a distributed, durable, ordered log. Unlike a queue that deletes consumed messages, it retains them, so multiple consumers read the same stream independently at their own offsets. One user event needs to update the feature profile, trigger a recommendation, feed logging, and maybe trigger retraining. Published once, consumed independently — and if a consumer crashes, it resumes where it left off with no events lost. This decoupling is what makes real-time AI systems maintainable.

Key Takeaways

Choose the right compute per workload (VMs for GPU training, containers for serving). Solve Kubernetes GPU scheduling before it surprises you. Design caching across all three layers. Defer microservices until organizational scale demands them. Put Kafka at the center of your event flow from day one if you expect real-time features.

None of this is glamorous. But every model serving millions of users sits on infrastructure decisions someone got right before the model was ever trained.
