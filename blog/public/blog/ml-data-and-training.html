AI System Design: Part-2 - From Raw Data to Trained Model: Engineering the ML Lifecycle

Why data pipelines are harder than model training — and why most production failures trace back to decisions made before a single gradient was computed.

Why data pipelines are harder than model training — and why most production failures trace back to decisions made before a single gradient was computed.

Ask experienced ML engineers where production failures start and they'll say the same thing: not in the model, in the data. A 2025 analysis found 60% of ML project failures trace back to data pipeline issues, and training-serving skew silently degrades 40% of production models.

This is the part textbooks skip because it isn't mathematically interesting. It's engineering, plumbing, and operational discipline — and it's where the real work lives.

The Mindset Shift: Data Is Code

A dataset isn't a static artifact — it's the output of a pipeline. Like any software, that pipeline has bugs, changes over time, and needs version control.

In traditional software, reproducing a bug means checking out the code at the right commit. In ML, reproducing a model means checking out the code and the exact dataset it trained on. Code alone reproduces the training process, not the trained artifact. This is what drives the whole discipline: making data pipelines reproducible, versioned, testable, and observable — just like code.

Data Ingestion: Where It Starts and Breaks

Batch ingestion runs on a schedule — simple to reason about, easy to rerun, fine when a few hours of staleness is acceptable. Stream ingestion processes events in real time, landing within seconds. The question isn't "which is better" but "what latency does my system actually need?" Real-time recommendations need streaming (behavior from 5 minutes ago is predictive); a weekly-retrained churn model doesn't. Defaulting to streaming "because it's modern" adds complexity that only pays off when latency genuinely demands it.

Schema validation is the unglamorous lifesaver. Upstream data is untrustworthy — a DB migration, a vendor API change, or a new app version can silently corrupt training data. Validate every record against expected types, ranges, and cardinalities at ingestion; failures alert rather than pass through. Great Expectations, Pandera, and dbt tests make this version-controlled. Missing a bad record is recoverable; discovering six weeks post-deploy that the model trained on corrupted data is not.

Lake, Warehouse, or Lakehouse?

A data lake stores raw data cheaply in any format — but queries are slow, with no transactions or versioning. A data warehouse stores structured data for fast analytical SQL with ACID semantics — but it's costly and poor for unstructured data. The Lakehouse (Delta Lake, Iceberg, Hudi) is the 2026 consensus: it adds transactions, schema enforcement, and versioning to object storage. Crucially, it allows time-travel queries (SELECT * FROM training_data VERSION AS OF '2025-09-01') — exactly what you need to reproduce a months-old training run.

Data Versioning: DVC and the Git Analogy

Git works for code because diffs are small. A 500GB dataset doesn't diff. DVC (Data Version Control) separates the data from its metadata: a small pointer file (content hash + remote location) lives in Git alongside your code, while the actual data lives in S3/GCS. git checkout a commit and DVC fetches the matching dataset version. dvc repro runs the full pipeline tracking which data, code, and parameters produced each artifact.

The alternative — training_data_v2_final_FINAL.csv in a shared bucket — is reality at most orgs until the painful day someone says "reproduce the model from three months ago" and nobody knows which dataset it used.

The Feature Store and Training-Serving Skew

A feature store computes, stores, and serves features centrally. The pitch sounds like tidiness, but the real reason it matters is training-serving skew.

Consider "purchases in the last 30 days." During training you write a SQL query over history. During serving you need that count in real time, in milliseconds, for one user. Two implementations of the same logic, by different people, in different environments — and they diverge. Maybe training uses UTC and serving uses local time. Maybe nulls are handled differently. Each is a silent bug: the model trains fine, validates fine, then quietly underperforms because serving inputs don't match the training distribution.

The key guarantee a feature store provides: the same transformation code feeds both the offline store (historical values for training) and the online store (latest values for serving). It runs once, not twice, so the two can't drift.

Scale is real: Uber's Michelangelo processes 10 trillion feature computations daily; Airbnb's Zipline serves sub-10ms; DoorDash's Fabricator cut feature engineering time 90%. Open-source: Feast (most deployed), Hopsworks, Feathr. Managed: Tecton, Databricks. Feast + Redis is a reasonable starting baseline.

Model Training: The Part Everyone Over-Focuses On

Experiment Tracking

The biggest training mistake is treating experiments as ephemeral. "I'll remember which run was best" is how you lose weeks. MLflow and Weights & Biases log everything per run: hyperparameters, metrics over time, hardware use, code version, dataset version, final artifact. You can query "all runs where validation AUC > 0.87" and compare. Every experiment becomes reproducible — and you can prove improvements are real, not artifacts of a changed dataset.

Distributed Training: When You Actually Need It

You need it when a model doesn't fit in one GPU's memory, or when single-GPU training is too slow for your iteration cycle. A 7B model in FP16 needs 14GB and fits on an 80GB A100; a 70B model doesn't — that's the memory threshold forcing model parallelism. For time: 30 days/iteration weekly means you need it; 4 hours means you don't.

Data parallelism replicates the model on each GPU, each processing a different data shard; gradients average via AllReduce. PyTorch DDP implements this — but communication overhead means ~1.5–1.8× on 2 GPUs, not a clean 2×. Model parallelism splits the model itself across GPUs (necessary when it won't fit), with pipeline parallelism (Megatron-LM) the most scalable form. DeepSpeed's ZeRO stages partition optimizer states, gradients, and parameters across GPUs, enabling hundred-billion-parameter training on commodity hardware.

Hyperparameter Optimization Beyond Grid Search

Manually trying learning rates wastes compute. Bayesian optimization (Optuna, W&B Sweeps) models the performance surface and picks the most promising next trial — often 5–10× fewer trials than random search. Hyperband/ASHA starts many trials with few epochs, kills the worst early, and reallocates to survivors — ideal when a single run is expensive. The practical stack: Ray Tune for orchestration, Optuna as the search backend, MLflow for logging.

The Pipeline That Ties It Together

Production ML is a DAG of automated jobs: Ingest → Validate → Transform → Features → Train → Evaluate → Register → Deploy.

Airflow is the most-deployed orchestrator for batch ML, though its scheduler was built for ETL, so GPU allocation needs careful Kubernetes integration. Kubeflow Pipelines and ZenML are purpose-built for ML; ZenML's swappable backends (local → Airflow → Kubeflow) are worth evaluating for new platforms. The key property: every step should be idempotent — rerunning from a checkpoint produces the same result. That requires careful checkpointing, deterministic shuffling (fixed seeds), and separating stateful from stateless operations.

What Architects Should Mandate

Three non-negotiables from day one: data versioning for all training datasets (DVC or Lakehouse time-travel) before the first training run; a feature store with online and offline planes before the first model using real-time features (retrofitting is far costlier); and experiment tracking that links code and data versions, not just metrics.

Unglamorous, no conference talk — but it's what separates teams that iterate reliably from teams that spend half their time debugging "why did this perform differently last time?"
