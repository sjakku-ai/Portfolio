AI System Design: Part-3 - Deploying and Keeping AI Models Alive in Production

Deployment is where confidence meets reality. Monitoring is where you find out how wrong you were.

Deployment is where confidence meets reality. Monitoring is where you find out how wrong you were.

Passing validation metrics is a milestone. Serving reliable predictions to real users, at scale, with acceptable latency, and still doing it six months later — that's the actual engineering challenge. This post covers the second half of the ML lifecycle: packaging and deploying without losing sleep, and detecting the slow, silent degradation that eventually claims every unmaintained model.

The Gap Between "Working" and "Deployed"

A trained model is a serialized file — weights and a graph. By itself it's inert. Deploying means building everything around it: a server that loads the model and serves requests, plus all the preprocessing that turns raw inputs into the exact tensors the model expects, and postprocessing back into your app's format.

The catch: training-time and serving-time preprocessing are usually written by different people, in different languages, at different times. They accumulate divergences. The model trained on inputs from a specific scaler; at serving the normalization is reimplemented slightly differently — and the model quietly produces bad predictions with no error thrown. This is why packaging matters.

Model Packaging

A deployed model should be self-contained: weights, transformation code, input schema, and library versions.

ONNX serializes the graph and preprocessing into one portable file that runs on ONNX Runtime regardless of training framework. MLflow format wraps the artifact with its flavor and environment, deployable to Databricks, SageMaker, or Azure ML. BentoML is more opinionated — you define a Service class with explicit schemas, and preprocessing lives in the same package so it can't diverge. Model registries add versioning and lifecycle governance (who approved each promotion) — essential when regulators demand audit trails.

Serving Patterns: Choosing the Right Shape

Online serving runs a persistent service responding in 100–500ms — right when a user is waiting (search ranking, fraud check at checkout). Infrastructure: TorchServe (clean PyTorch integration, weak LLM support), Triton (industrial-strength GPU serving with dynamic batching), and vLLM (the LLM standard, whose PagedAttention enables continuous batching).

Batch serving processes a dataset on a schedule, no latency pressure — right for churn scores computed nightly. Often far cheaper than GPU inference. The common mistake is defaulting to online "because it's modern"; batch is the cheaper, correct answer for a large fraction of use cases.

Streaming serving consumes events from Kafka and publishes predictions continuously — fraud detection, real-time moderation, IoT anomalies. The challenge is memory: models must fit the stream processor's executor and can't hot-reload without disruption.

Deployment Strategies: Limiting the Blast Radius

Every deployment is a risk — a model that aced your eval set can misbehave on full production traffic.

Blue-green keeps two identical environments and switches traffic atomically — instant rollback, but double the infrastructure during deployment. Canary routes a growing slice (5% → 25% → …) to the new model while monitoring each stage — the most common strategy: limited blast radius, real traffic for evaluation, no double infrastructure. Shadow sends all traffic to the new model but serves none of its predictions — zero user risk, complete comparison data, but expensive and slow.

The Silent Failure Problem

Take this seriously: 91% of ML models degrade over time, and 73% of production AI failures link to unforeseen input shifts. "Silent" is what makes it dangerous — unlike a crash, a degrading model produces normal-looking outputs. No alert fires. It just gets quietly wronger.

Case study. A bank trained a credit model on 2021–2023 data; validation accuracy 95%. Deployed January 2024. By September, accuracy had dropped to 87% — missing 8% more defaults. Interest rates had risen sharply; debt-to-income and savings distributions shifted; the model learned a low-rate world and applied it in a high-rate one. No code changed. It just stopped being right.

Data Drift vs. Concept Drift

Data drift is when feature distributions shift from training. Detect by comparing against baselines: KS test (continuous), Chi-square (categorical), Population Stability Index (PSI > 0.25 flags meaningful drift). Concept drift is when the input-output relationship itself changes — what counted as fraud in 2023 differs in 2024. Harder to detect because it needs labels; for delayed feedback, collect labels after the fact and evaluate on a rolling window, triggering retraining when metrics fall below threshold.

What Monitoring Actually Looks Like

Three layers, all necessary. Infrastructure: latency (P50/P95/P99), throughput, error rates, GPU/memory — Prometheus + Grafana, or Datadog. Model health: prediction distributions, per-feature drift, input quality, confidence — EvidentlyAI, Arize, WhyLabs. Business metrics: the downstream number the model should move (CTR, conversion, default rate). Infra and model metrics can look fine while the business metric declines — always trace changes to outcomes.

Closing the Feedback Loop

Monitoring without retraining is just observability. Fully automating this loop — including evaluation and deployment — is continuous training (CT). It requires automated eval harnesses (slice-based, not just aggregate accuracy), deployment pipelines, and alerting that distinguishes "degraded naturally, retraining underway" from "something is broken, page on-call."

Explainability: The Accountability Layer

In regulated domains — lending, hiring, healthcare, insurance — "why did the model predict this?" is a legal requirement. SHAP is the most theoretically grounded, assigning each feature a game-theoretic contribution. Model-agnostic but expensive for large ensembles. LIME trains a simple local surrogate — faster but less consistent. For LLMs, explainability means which retrieved chunks influenced the answer and whether output is grounded in context.

A practical pattern: compute SHAP for high-risk predictions only, store them in the audit log, and surface them to human reviewers — satisfying audits without adding serving latency.

The Monitoring Trap

Teams either monitor too much (40 metrics nobody reads) or too little (just latency). The pragmatic start: PSI for your top 10 features (daily), performance on a rolling 7-day labeled window, and P99 latency per model version. Monitor those three reliably and you'll catch 80% of problems. The goal isn't maximum observability — it's actionable observability, where every alert has an owner and a documented response.
