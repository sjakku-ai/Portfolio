Building a GPT From Scratch, Part 1: Why Build It Instead of Just Calling an API?

Starting a new series: 20–30 minutes a day building a mini GPT-like chatbot from the ground up, to actually understand the tokenization → inference → decoding pipeline instead of just knowing how to call it.

It’s never been easier to add an LLM to a project — import a client, drop in an API key, send a prompt, get text back. That’s the right call for almost every real product. But it also means it’s possible to build AI features for years without ever knowing what happens between the prompt going in and the text coming out.

I’d rather not be in that position. So instead of calling someone else’s model, I’m building a small one myself — a mini GPT-like chatbot, from scratch, piece by piece. I believe, the fastest way to actually understand how a language model works is to build a working, if tiny, version of one.

This post is the kickoff: what I’m building, why, and the four-part roadmap I’ll be working through in public, 20–30 minutes at a time.

The plan, in plain English

Stripped of jargon, an LLM chatbot is four problems stacked on top of each other:

  1. Getting the text to train with — data collection. A model can’t predict language it’s never seen.
  2. Turning words into numbers — tokenization. Models don’t read text; they do math on numbers, so the text has to become numbers first.
  3. Getting the AI to predict the next word — text generation. This is the actual “model” part: given everything so far, what comes next?
  4. Turning it into a real app — wrapping the model in something a person can actually use.

Each of those is its own rabbit hole, and I’m planning to spend real time in each one rather than rushing to a demo. The end goal is a working “Custom GPT” web app that can summarize content from any URL I feed it — a small, concrete product, built on a real (if small) understanding of the pipeline that powers tools like ChatGPT, rather than just the skill of prompting one.

Why bother, when the APIs are right there

Because using an API and understanding the system are different skills, and only one of them transfers when something breaks. When a hosted model does something unexpected — truncates output, hallucinates, gets the context window wrong, costs more than expected — the people who can actually reason about why are the ones who understand what’s happening under the hood: how tokenization works, why generation is autoregressive, what decoding strategy is doing to the output. Calling an API teaches you the interface. Building the pipeline teaches you the system.

The goal isn’t to out-build GPT-4. It’s to be able to open up any LLM-powered system and know exactly what every layer is doing.
